# gepa-real-2026-06-27 — first real GEPA shadow campaign

The first campaign whose candidate was generated by the **real GEPA optimizer** (not the deterministic
control backend). It promotes nothing.

## Setup

GEPA needs an LM for its reflective proposer. It was wired to a **local** model — no credentials, no
external egress:

```bash
python3 -m venv /tmp/gepa-venv
/tmp/gepa-venv/bin/pip install gepa litellm
GEPA_REFLECTION_LM="ollama/qwen3:8b" /tmp/gepa-venv/bin/python tools/run-gepa-real.py \
  --campaign evals/gepa-shadow/campaigns/gepa-real-2026-06-27/campaign-manifest.json \
  --backend gepa --max-metric-calls 8
```

`tools/run-gepa-real.py` reads `GEPA_REFLECTION_LM` (a litellm model name) and builds the reflection LM
via `gepa.optimize_anything.make_litellm_lm`. The name is a model identifier, not a secret, and is
recorded in each candidate's `gepa-trace.json`. The venv is intentionally **not** committed; GEPA and
litellm are not harness runtime dependencies.

## What happened (the finding)

From an **empty seed** (new-technique-card track), the unconstrained 8B proposer drifted hard
**off-scope**: it generated a binary-exploitation card (`cand-gepa-new-technique-card-001` —
"Memory Corruption via Double-Free", ASLR bypass, gadget chains, CVE references) with malformed
front-matter, nothing to do with this harness's LLM-red-team scope.

| candidate | structural gate | measured materialization | disposition |
| --- | --- | --- | --- |
| cand-baseline (no-op) | allow | allow (conformance ✓) | probe |
| cand-gepa-new-technique-card-001 (GEPA) | allow | **block (conformance ✗)** | **NOT PROMOTABLE** |
| cand-gepa-invalid-gold-touch (control) | block (preflight) | block | block |

The structural contract gate passed the GEPA card (a single new file inside the mutable allowlist). The
**measured materialization boundary (Phase 10H0)** applied it in isolation and ran trusted conformance,
which **blocked** it — so the promotion bundle is NOT PROMOTABLE. The deterministic shadow scoreboard
(which never reads card content) labels it `probe`; the authoritative answer is the materialized `block`.
That is exactly why 10H0 made applied repository state, not the structural gate, the authority.

## Honesty

Nothing was applied to Plane 1 (`skills/` is untouched). No auto-promotion. No secrets in artifacts. The
off-scope card is retained only as **evidence of proposer drift that the gates caught** — it is malformed,
non-operational, and blocked. `autonomous_gap_closure_count` stays **0**: GEPA found no in-scope
improvement, and a blocked off-scope card is not a discovered-and-closed gap.
