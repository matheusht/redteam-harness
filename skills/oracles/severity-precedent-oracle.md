# Severity-precedent oracle — "what does disclosed-CVE precedent say the score is?"

The severity-methodology pass. Given a finding whose validity is already settled, it supplies
per-class CVSS scoring precedent keyed to the metrics people actually argue about, so a report's
Severity section matches how real scorers have scored the same bug shape — instead of defending a
vector from metric definitions and first principles alone. This card is the terse, invocable-by-id
version; the full adjudicated distribution (per-metric counts, every CVE, the discriminators) lives in
`skills/oracles/references/severity-precedent-corpus.md`.

## What this is / is NOT

- **IS:** a severity-only precedent reviewer. It answers "for this bug class, which CVSS values are
  convention, which are genuinely contested, and which real CVEs are the closest comparables to cite?"
- **IS NOT** a validity pass. It does not decide whether the finding is real, reachable, or correctly
  attributed — that is `regrade-oracle` (`engine/regrade.md`).
- **IS NOT** a PoC-fidelity pass. It does not decide whether a specific artifact proves the claim —
  that is `pocinator-oracle` (`engine/pocinator.md`).
- Runs **later** than both, at severity-authoring / report-writing time, and only on a finding those
  passes have already cleared. `engine/regrade.md` §5 carves this out explicitly as a "narrower,
  later, severity-methodology-only pass."

## How to use it

1. Take the confirmed finding's **CWE + mechanism** and find its bucket in the reference table below
   (match on mechanism, not just the CWE label — CWE strings are noisy).
2. Confirm which **CVSS version** the target program's calculator actually uses (do not assume it
   transfers across programs), then read that version's **typical vector → score**. The 3.1 and 4.0
   distributions are counted and gated **separately** — never merge a metric across versions.
3. For each **contested** metric, apply the named **discriminator** to your specific finding to pick
   the value, and state the split in the report rather than asserting a single number.
4. **Cite the closest listed CVEs** as precedent for the values you land on.
5. **Match precedent by default.** Depart from it only with an explicit argument grounded in the shown
   distribution (e.g. "this finding is the boundary-crossing case the S:C minority covers, here is the
   concrete pivot") — not from metric definitions alone. A departure with no distributional backing is
   exactly the novel-scoring move this oracle exists to prevent.

## The gate: IN / CONTESTED / THIN

Each metric in a bucket carries one verdict, decided per CVSS version:

- **IN (convention — assert it):** ≥6 verified CVEs in that version's pool **and** a clear ~2/3 mode
  **and** no intra-CVE authority disagreement on that metric. State the modal value as the convention,
  naming any fact-driven minority and its discriminator.
- **CONTESTED (name the split, don't assert):** a structured / bimodal split with a nameable
  discriminator, **or** documented intra-CVE authority disagreement on that exact metric (NVD vs CNA,
  same version) — either one forces CONTESTED **regardless of the mode**. A CONTESTED verdict is a
  success of the gate, not a failure: it hands the report an honest split plus the rule for picking a
  side, which is more defensible to a triager than a false convention.
- **THIN (illustrative only — assert nothing):** fewer than 6 verified CVEs in that version's pool.
  Values are reported to illustrate shape, never as a convention. Every bucket's CVSS 4.0 sub-pool is
  thin unless stated otherwise; do not carry a firm 3.1 convention across into 4.0.

## Reference table (keyed by bug class → CWE)

Format per bucket:
`3.1 typical vector → score  |  4.0 typical vector → score` · contested/held focus metrics with
value `(k/n)` and discriminator · closest CVEs to cite · `n = 3.1:x / 4.0:y`.

- **Contested / focus metrics:**
  - **CVSS 3.1 Scope (S)** — CONTESTED: S:U dominant in primary scoring (12/12 NVD/primary vectors) but a genuine structured dissent: NVD and most CNAs score same-authorization-context cross-object IDOR as S:U, while some CNAs escalate to S:C when the pivot crosses a tenant/domain/org boundary (CVE-2023-38051 CNA scored S:C; CVE-2023-44254 CNA scored S:C). ### Auth bypass (generic mechanism / middleware / gateway) (CWE-287 / CWE-863)
- **Typical vector:** 3.1 `CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H -> 9.8 (modal archetype; 3 exact matches: CVE-2022-1388, CVE-2022-40684, CVE-2023-22515). Held metrics: AC:L, PR:N, UI:N, C:H. Scope and Integrity are CONTESTED, so the S:U and I:H components of this vector are convention-of-central-tendency, not asserted conventions.`  |  4.0 `thin`
- **Contested / focus metrics:**
  - **CVSS 3.1 Scope (S)** — CONTESTED: S:U 6/9, S:C 3/9 — mode is exactly ~2/3 but a genuine same-version authority disagreement is present — Discriminator: S:U when the bypass only reaches resources inside the same protected component/service (Next.js/Fortinet/Kong endpoints, Confluence per NVD); S:C when impact crosses a security-authority boundary — host-level user creation (CVE-2023-20198, S:C, 10.0), a separate AuthZ plugin being defeated (CVE-2026-34040 per CNA), or control of another user's device (CVE-2026-30368). Intra-CVE authority split is explicit: CVE-2026-34040 NVD S:U=7.8 vs CNA S:C=8.8 (same 3.1 version), and CVE-2023-22515 NVD S:U vs Atlassian S:C (cross 3.0/3.1). Per the gate, same-version authority disagreement forces CONTESTED regardless of the 2/3 mode.
  - **CVSS 3.1 Confidentiality (C)** — IN: typical C:H (8/9) — Only outlier is CVE-2026-30368 (C:L) — a scoped client-side device-token abuse. 8/9 = 0.89 >= 2/3, no authority contest.
  - **CVSS 3.1 Integrity (I)** — CONTESTED: I:H 6/9, I:N 2/9, I:L 1/9 — mode exactly ~2/3 but a structured (not noise) split — Discriminator: I:H when a state-changing/privileged action is demonstrated (admin-account creation, admin-API write, config/SSH-key change — CVE-2022-1388/40684/2023-22515/20198/2026-34040); I:N when only unauthenticated read/reach of a protected resource is shown with no write demonstrated — Kong JWT-plugin bypass (CVE-2021-27306) and Quarkus path-auth static-resource reach (CVE-2026-50559). The two I:N cases map exactly onto the read-only-reach subclass, making this a structured bimodal split rather than a clean mode.
  - **CVSS 3.1 Attack Complexity (AC)** — IN: typical AC:L (8/9) — Only outlier CVE-2026-30368 (AC:H) requires target email + district ID plus service-worker WASM manipulation. 8/9 >= 2/3.
  - **CVSS 3.1 Privileges Required (PR)** — IN: typical PR:N (8/9) — Only outlier CVE-2026-34040 (PR:L, AV:L — local low-priv Docker context). 8/9 >= 2/3; the bucket is overwhelmingly pre-auth.
  - **CVSS 3.1 User Interaction (UI)** — IN: typical UI:N (9/9) — Unanimous. These are direct unauthenticated / server-side bypasses; no victim interaction on any vector of record.
  - **CVSS 4.0 Vulnerable-System Confidentiality/Integrity (VC/VI)** — THIN: VC:H 3/3, VI:H 3/3 (illustrative, n=3) — Consistent high vuln-system C/I, but n=3 is below the 6-CVE gate — corroborated by the three shadow NVD 3.1 vectors (all C:H/I:H) yet no convention asserted.
  - **CVSS 4.0 Vulnerable-System Availability (VA)** — THIN: VA:H 1/3 (CVE-2024-0012), VA:N 2/3 (illustrative) — Too few and split; no convention.

### CSRF (CWE-352)
- **Typical vector:** 3.1 `AV:N/AC:L/PR:N/UI:R/S:U/C:H/I:H/A:H = 8.8 HIGH (modal NVD/primary reconstruction; anchored by CVE-2024-39022 and CVE-2023-23724 which score exactly this. - **Contested / focus metrics:**
  - **CVSS 3.1 Scope (S)** — IN: S:U (8/8, 100%) — No CSRF in the bucket crossed a security-authority boundary; every case (incl. the RCE-adjacent OFBiz-class chains scored under 3.1) kept S:U. Convention: S:U unless the forged request pivots into a distinct component/authority. No authority disagreement on Scope anywhere in the bucket.
  - **CVSS 3.1 User Interaction (UI)** — IN: UI:R (8/8, 100%) — Victim must visit the malicious page — inherent to classic CSRF. 3.1's binary N/R scheme makes this unambiguous, unlike 4.0 where the same phenomenon splits N/P/A (see below).
  - **CVSS 3.1 Privileges Required (PR)** — IN: PR:N (7/8, 87.5%) — Attacker holds no privileges; the victim's authenticated session supplies them. Lone PR:L outlier CVE-2024-0555 (WIC1200) where the lured actor is modeled as an already-authenticated low-priv user.

Discriminator: C:H when the forged action yields admin/session compromise or reads back sensitive data (7534, 3589, 0555, 23724, 39022); C:N when it is a pure state-changing write with no read-back (22811 config override, 44342 OAuth account-bind); C:L intermediate (3083). — Compounded by intra-CVE authority disagreement: secondary CNAs push C:N even lower on impact (INCIBE C:N on 0555, Patchstack C:N on 23724).   - **CVSS 3.1 Integrity (I)** — CONTESTED: I:H unanimous across re-verified primary/NVD vectors (8/8) — the core CSRF impact — BUT systematic intra-CVE authority disagreement drops it to I:L on 3/8: CISA-ADP (7534), INCIBE (7534 A only; 0555 I:L), Patchstack (23724 I:L). Integrity, not Confidentiality, is the metric CSRF fundamentally implicates.
  - **CVSS 4.0 Attack Requirements (AT)** — IN: AT:N (5/7, 71%) — 2 AT:P outliers (CVE-2024-43684, CVE-2024-48962) — the SameSite-bypass / race-condition chained cases. Convention: AT:N for plain CSRF; AT:P only when a specific execution condition (timing/SameSite bypass) must hold.
  - **CVSS 4.0 Attack Complexity (AC)** — IN: AC:L (5/7, 71%) — Same 2 AC:H outliers (43684, 48962) as AT:P — chained/defense-bypass cases. Mirrors the 3.1 AC:L convention.

  - **CVSS 4.0 User Interaction (UI)** — CONTESTED: UI:A (3/7), UI:P (2/7), UI:N (2/7) — no mode; a 4.0-specific modeling contest with no consensus. Discriminator: UI:A when scored as victim must actively act inside the app for the forgery to land (48962, 32989, 24345); UI:P when merely loading a malicious page suffices — the technically-correct CSRF model (43684, 27146); UI:N when scored login-only (47846, 51485), arguably a mis-model of CSRF. — This is the single biggest 4.0 scoring ambiguity for CSRF: the clean 3.1 UI:R (8/8) fractures into N/P/A. ## Boundary + honest limitation

- **Precedent is convention, not correctness.** This oracle reports how real scorers *have* scored a bug
  shape, not the "true" CVSS. Matching precedent is the defensible default and what a triager expects;
  it is not a proof that the modal vector is right.
- **The corpus is skewed by the NVD backlog** — CVSS 4.0 sub-pools are thin across almost every bucket,
  so most 4.0 guidance is illustrative, not convention. Confirm the target program's actual CVSS version
  and never carry a 3.1 convention across into 4.0.
- **CWE labels are imperfect** — authorities disagree on labels for identical bugs (e.g. CWE-88 vs
  CWE-1321, CWE-787 vs CWE-120). Match on mechanism, and use the closest-CVE list, not the CWE string
  alone, to pick a bucket.
- **A CONTESTED verdict is a success of the gate, not a failure.** It means the honest answer is "scorers
  split, here is the discriminator" — which belongs in the report as a stated split, not papered over
  with a false single number.
- Full per-metric counts, every CVE, and the complete discriminator set:
  `skills/oracles/references/severity-precedent-corpus.md`.
