Architecture Brainstorm: GEPA, DSPy, and Agentic Red-Teaming Harness

A practical review of where reflective optimization belongs in the current harness, when DSPy becomes useful, and which agentic security-system ideas should remain gated.

Executive recommendation

Add GEPA first as an offline, human-gated optimizer for technique-card variants. Keep DSPy out of the core path for now. Treat Cloudflare- and Depthfirst-style agentic systems as directionally useful, but sequence them behind evaluator maturity, replay, hermetic benchmarks, and scope controls.

01Executive Recommendation

Recommended path

Implement GEPA as the first real autoresearch mechanism, but only as a shadow campaign that proposes technique-card variants and emits an evidence bundle.

Why

GEPA optimizes textual components using execution traces and evaluator feedback. That maps directly to this harness's casebooks, scorers, cost ledgers, and false-discovery gates.

Not yet

Do not let an optimizer mutate oracles, casebooks, routing gold, scorers, budgets, sealed data, or live target behavior. DSPy is a later adapter spike, not a prerequisite.

Practical decision: GEPA belongs outside Plane 1. It proposes; the evaluator blocks or allows; a human promotes. That keeps the optimizer from grading itself.

03GEPA Fit

What GEPA should optimize

  • Technique-card variants.
  • Hunter brief phrasing.
  • Benign ladder sequencing hints.
  • Bounded continuation and reframing variants.

What GEPA must not optimize first

  • Oracles, scorers, gold labels, budgets, casebooks, or sealed data.
  • Vulnerability objectives or safe_signal definitions.
  • Live engagement traffic or target-specific strategy.
  • Automatic Plane-1 promotion.

Inputs and outputs

Side Artifacts Architectural note
Input Candidate technique, campaign manifest, frozen evaluator set, traces, scorer output, cost ledger, replay outcomes. Every input that affects acceptance must be hashed in the campaign manifest.
Output Candidate diff, lineage, failure categories, keep/discard evidence, promotion bundle. The output is evidence, not a direct edit to skills/.
Gate Conformance, false-discovery, routing/hermetic eval, replay stability, protected-case regression check. False-discovery increase is a hard veto, not a score tradeoff.

Minimal first implementation

  1. Create a GEPA shadow campaign runner.
  2. Let it mutate one declared technique variant.
  3. Evaluate against frozen fixtures and hermetic targets.
  4. Emit an evidence bundle and proposed diff.
  5. Require human review before any Plane-1 promotion.

04DSPy Fit

DSPy is useful when the harness has stable LM modules with typed inputs and outputs. Today, the working core is deliberately simple: markdown cards, JSON schemas, stdlib scorers, and deterministic gates. That makes DSPy interesting, but not required for GEPA.

Option Benefits Costs Recommendation
GEPA-only custom adapter Smallest integration; preserves current card/evaluator architecture. Requires custom trace formatting and adapter code. Build now
GEPA + DSPy spike Typed predictor traces and cleaner LM module boundaries. Introduces dependency and abstraction layer. Spike later
Full DSPy rewrite Coherent typed LM-program architecture. High migration cost; rewrites working harness pieces prematurely. Do not do now

DSPy becomes attractive after the harness has multiple stable LM programs. It is not needed for the first GEPA shadow campaign.

07Proposed Architecture Changes

Must-have now

Item Description Why it matters Implementation surface Risk Acceptance criteria
GEPA shadow campaign adapter Evaluate one technique variant against frozen fixtures and return score plus feedback. Smallest useful GEPA integration. evals/gepa-shadow/, tools/run-gepa-shadow.py Scope creep into evaluator mutation. Only one declared technique changes; bundle only, no Plane-1 edit.
Candidate and campaign manifests Record mutable component, parent hash, evaluator hashes, budgets, model settings, lineage. Makes runs replayable and tamper-detectable. New manifest schemas and conformance checks. Over-schema early. Budget/evaluator/gold/candidate hash mismatch blocks.
Rotated casebook Add a fresh sanitized case not derived from the original live run. Prevents original-engagement-shaped optimization. casebooks/rotation-case-01-web-api/ Public-source contamination. Includes positives and held/refuted rows; no raw flags, names, or secrets.
GEPA hard-gate wrapper Run conformance, false-discovery, routing/hermetic eval, replay, and protected-case checks. Prevents optimizer from bypassing rigor. Campaign runner reusing existing tools. Scalar-score gaming. FDR increase blocks; equal coverage requires at least 10% cost reduction.

Should-have next

Item Description Why it matters Implementation surface Risk Acceptance criteria
Canned/fake-model hermetic targets Deterministic targets for LLM05, LLM06, LLM07, and router differential. Real LLMs drift; hermetic benchmarks need stable answers. evals/hermetic/targets/ Overfitting to toy transcripts. Each target has confirmed and held/refuted cases.
Task registry Append-only record of campaigns, tasks, attempts, artifacts, and retries. Persistence before parallelism. SQLite or JSONL run log. Premature platforming. Failed run can resume without losing completed evaluations.
Deterministic reporting Generate reports from structured findings and evidence refs. Prevents invented severity, impact, or proof. tools/render-report.py Plain initial reports. Every report claim links back to finding fields or evidence refs.
Vulnerable app lab catalog Reviewed list of possible AI/web vulnerable apps for future targets. Benchmark diversity without Plane-1 pollution. evals/lab-catalog.md Unsafe objectives or raw challenge import. Each candidate has license, safety, contamination, and mapping notes.

Future expansion

DSPy module spike

Test one isolated typed LM module for routing-output or evaluator-output normalization. It must outperform or simplify the existing prompt+JSON path before becoming core.

Gapfill planner

Generate bounded tasks for under-tested coverage cells. Needs task registry and budget controls before it is safe.

Dedup clustering

Cluster by dedup key and root cause before reporting or scoring. Prevents duplicate coverage inflation.

Out of scope / requires review

  • GEPA optimizing oracles, scorers, gold labels, casebooks, budgets, or sealed data.
  • GEPA running directly against live engagements.
  • Autonomous exploit-chain construction beyond benign canaries and approved hermetic labs.
  • Fleet-scale multi-repo tracing before persistence, dedup, and validation are stable.

08Suggested Implementation Sequence

Phase Work Exit criterion
Phase 1: smallest useful GEPA integration Add campaign/candidate manifests, GEPA custom adapter, one mutable technique variant, frozen eval run, and evidence bundle output. Candidate can be generated, evaluated, blocked or kept, and replayed without touching Plane 1.
Phase 2: evaluation and benchmark loop Add rotated casebook, fake-model hermetic targets, replay stability, validation split, and expanded false-discovery/adversarial corpora. Candidate cannot pass by increasing false positives, changing budget, or regressing protected cases.
Phase 3: agentic harness expansion Add task registry, gapfill planner, deterministic reporting, dedup clustering, and optional DSPy module spike. Longer campaigns become resumable, bounded, and auditable.
Phase 4: robust security-agent style system Add independent validation model, multi-agent recon/hunt/validate stages, persistent coverage matrix, and broader hermetic labs. Harness resembles a constrained security-agent system while preserving scope and evidence gates.

09Accepted Decisions

Question Decision Why
First GEPA campaign target Run a multi-track campaign over task-reframing, decomposition, and one new technique-card candidate. Campaign breadth is useful, but each candidate still mutates one declared component for attribution.
Need another rotated case first? No for shadow runs; yes before promotion. The integration can start on current fixtures, but Plane-1 promotion needs broader evidence.
Candidate format Use JSON manifest + JSON scores + Markdown evidence bundle + diff. Machines gate JSON; humans and models review Markdown; patch provenance lives in the diff.
Standalone GEPA or DSPy first? Standalone GEPA adapter first. DSPy is a later bounded spike. The current harness is markdown cards, JSON contracts, and deterministic scorers; DSPy should earn its place.
Next rotated-case sources Use sanitized archetypes first; Dockerized targets later for hermetic benchmark expansion. Casebooks are cheaper, easier to scrub, and better for laptop-friendly qualification.
Promotion PR-only promotion after scorers, replay, bundle, and human review. Manual copy loses provenance; direct-to-main bypasses the evidence chain.

10Implementation Plan

The detailed implementation plan lives in docs/gepa-autoresearch-implementation-plan.md. The sequence below is the architectural spine.

Phase Build Exit criterion
0 — alignment Roadmap, decision record, and artifact updated with accepted GEPA/DSPy decisions. GEPA first, DSPy later, PR-only promotion, and review-required expansions are explicit.
1 — contracts Campaign manifest, candidate manifest, promotion bundle, and hash checks. Undeclared mutations or evaluator/budget/gold drift block.
2 — GEPA shadow runner Standalone GEPA adapter, multi-track campaign, no-op baseline, candidate artifacts. GEPA can propose and evaluate candidates without touching Plane 1.
3 — scoring policy Candidate scorer with allow, block, and probe. FDR regression blocks; duplicate coverage does not count; cost rule is enforced.
4 — replay Primary run plus two fresh-session replays. Unstable wins become probe, not allow.
5 — promotion bundle Bundle renderer and PR checklist. Successful candidates promote only through reviewed PRs.
6 — benchmark expansion Fake-model hermetic targets, second rotated case, SSRF decision. Evaluator breadth improves before real ratchet pressure increases.
7 — DSPy spike One isolated routing/evaluator normalization module if useful. DSPy shows measurable value or stays out.
8 — standalone adapter GEPA adapter surface, generated candidate artifacts, full shadow scoring/replay/bundle chain. Adapter produces scoped deltas and promotes nothing; candidate-applied evaluation remains next.
Status update, 2026-06-26: phases 0–8 now exist in shadow. The local Phase 8 run used the deterministic backend because GEPA is not installed/configured in this environment, while tools/run-gepa-real.py --backend gepa is wired for the standalone GEPA optimize_anything API. DSPy was spiked and shelved. The next honest build is candidate-applied evaluation: applying one candidate diff in an isolated workspace/session and measuring behavior against the same frozen evaluator.

11Future Review Tracks

Live GEPA

Start as advisory mode over scrubbed traces. Bounded and autonomous live modes require run registry, scope enforcement, rate limits, and explicit review.

Evaluator co-evolution

GEPA may eventually propose oracle/scorer/gold changes, but only behind a sealed meta-evaluator and protected historical cases.

PoC ladder

Evidence explanation, benign canary proof, local hermetic PoC, scoped live PoC with human confirmation, then autonomous chaining only after separate review.
