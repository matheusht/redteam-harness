Design decisions where the spec was ambiguous

Pattern cards are a first-class type: pattern, not vuln cards

The plan said "use the existing SKILL.md card format." I made pattern cards a distinct card type with an activation front-matter block (strong/weak/negative) and a routes_to field, and documented the type in schemas/technique.schema.md. Rationale: a pattern is guidance the orchestrator routes to, not an objective it runs, so it deliberately omits safe_signal/impact_escalation — but the doc states it still inherits the full oracle discipline so the omission can't be read as a loophole.

Casebooks live at repo root casebooks/

Matches the plan's stated default. Kept the three structured files under JSON Schema and left hypotheses.yml / lessons.md / promotion-notes.md as light prose/YAML, documented in casebooks/_SCHEMA/README.md. v1 goal is repeatable sanitized learning, not exhaustive schema coverage.

Routing got its own doc, not only inline edits

Plan change 5 said "modify AGENTS.md / loop.md lightly." I did the light edits and added engine/routing.md. Reason: the routing step ("recon emits activation signals before card selection") is the conceptual heart of "the LLM chooses" — burying it as a clause in two runbooks would under-weight it. The runbook edits now point at the doc. Edits to AGENTS.md and CLAUDE.md are identical (they're twins) to avoid desync.

Field names kept as routing teeth

The casebook retains API field names (ownerId-style selector, modelAi, mandatoryTools) and the v1/v2 route shape, because those are the activation signal — stripping them would gut the routing value. Everything that is actually a secret (ids, tokens, canary string, passwords, signed URLs, screenshots) is stripped. The sanitization attestation in case.yml records exactly what was removed.

Deviations intentional departures

  • Added a card-contract section to schemas/technique.schema.md documenting type: pattern + activation. Not in the plan, but the new type is incoherent undocumented. Kept to one block.
  • Edited CLAUDE.md too, not just AGENTS.md. The plan named AGENTS.md; CLAUDE.md is its twin and would have drifted. Both now carry the same recon→routing step.
  • Pointer line added to engine/loop.md (a two-line blockquote) so the loop explicitly runs the objectives routing surfaced, rather than a fixed card list. Minor, additive.
  • ROADMAP not yet edited. Left untouched in this pass to keep the diff scoped to the plan; flagged as an open question below.

GEPA autoresearch — Phase 3 (keep/discard) rev 4 · 2026-06-26

Implementing docs/gepa-autoresearch-implementation-plan.md Phases 3–4. Phases 0–2 landed earlier (commits 1c7d4ab docs/roadmap, b4b8b4a campaign/candidate contracts + gate, 6fcc780 shadow runner). This section is the running record for the scoring policy and the replay gate.

The framing that colors all of Phases 3–4. Shadow mode has no model in the loop yet. With no GEPA-generated behavioral change to measure, every gate-passing candidate reproduces the incumbent scoreboard — so the honest verdict is probe (eligible, not better) and replays are trivially stable. The machinery is proven; a winning candidate is not claimed. Real deltas arrive when the GEPA adapter is wired (post-Phase-5).

Acceptance, mapped to evidence

Spec acceptanceEnforced where
False-discovery regression is always blockscore-candidate.decide() hard veto on false_discovery_rate>0; self-test "FDR regression → block even with new coverage"

Design decisions spec ambiguous

  • Coverage = set of dedup_key strings. The plan says "distinct clean confirmed coverage case" without fixing a representation. I used the dedup_key space finding.schema.json already defines, so "distinct" is a literal set difference and duplicate coverage contributes zero. Novelty is therefore only as good as dedup_key quality.
  • Cost = model_calls + target_calls, equal weight. "Reduce cost by ≥10%" named no unit. I used the call-count ledger already in the evidence contract's run_cost; it is stable and model-independent. Tokens stay recorded but are excluded from the ratio.
  • "discard or return probe" → always probe. An eligible-but-not-better candidate is kept as probe, never silently discarded; discard is a human action in PR review.
  • Shadow eligibility flags derived from the gate. For a gate-passing shadow candidate, conformance_passed / evidence_contract_complete / budget_unchanged / single_mutation are set true — the contract gate already enforces budget/single-mutation/bundle, and the tree is conformance-green in CI. Must be sourced per-candidate once a model generates real diffs (see Q).

Deviations intentional

  • Block strictly dominates utility. Every eligibility veto is computed before the utility rule and short-circuits to block even when the candidate also adds coverage. Faithful reading of "mechanical failures cannot be overridden," but stricter than the bare utility text.
  • score-candidate integrated into the shadow runner. The plan lists it standalone; I also wired it into run-gepa-shadow.py so the report shows a live keep/discard verdict per candidate. The pure function stays independently testable; the integration is additive.

Surface built: tools/replay-candidate.py (pure assess() + 8-case --self-test + a --campaign generator), replays/ artifacts per allowed candidate, a campaign replay-report.{md,json}, and a Replay section added to each allowed candidate's evidence bundle.

Acceptance, mapped to evidence

Spec acceptanceEnforced where

Claimed new coverage reproducescoverage_stable: claimed set ⊆ every run's coverage; self-test "replay loses claimed coverage → probe"

Replay artifacts included in the evidence bundleeach allowed candidate's evidence-bundle.md has a Replay section pointing at replays/{primary,replay-1,replay-2,replay-summary}.json

Design decisions spec ambiguous

  • The replay gate can only downgrade. The plan says non-deterministic success becomes probe. I made the rule total: block→block, allow→allow|probe, probe→probe. Replay never upgrades a verdict (a flaky candidate cannot be promoted, and a replay cannot rescue a blocked one).
  • Policy violations are treated as instability. Fewer than two replays, a benchmark-version drift, or a budget change set policy_ok=false, which forces an allow down to probe — the same effect as an unreproduced claim. "Identical benchmark version and budget" is thus enforced, not assumed.
  • benchmark_version = hash of the frozen-input hashes. The plan says "identical benchmark versions" without defining a version. I derived it from the campaign's frozen_inputs sha256 set, so the version literally is the frozen benchmark; any drift the contract gate would catch also changes the replay version.
  • Recorded session identifiers are deterministic. session_id = <candidate>-session-<i>, fresh_session true for replays. No RNG — the harness forbids Date.now()/random in scripts and determinism keeps replays auditable.

Deviations intentional

  • Replays generated by re-running the deterministic benchmark, not a live model. In shadow mode the three runs are identical reproductions (cost spread 0), so the artifacts are real and the gate is exercised, but no variance is manufactured. The self-test injects real instability to prove the downgrade fires — the campaign data does not fake it.
  • Final verdict (Phase 3 ∧ Phase 4) recorded in replay-summary.json, not folded back into the shadow runner's report.json. Keeps the Phase-3 report immutable and the replay-adjusted verdict separately auditable; a later phase can join them for the promotion bundle.

Design decisions spec ambiguous

The spec lists four targets; I built a single runner with a per-oracle derive() and a uniform adjudication (leaking primary + firing positive control + clean negatives), mirroring the BOLA harness. Each request carries a recorded model_output (or dispatch/policy fields) and an expect; the runner derives (signal, leak) and flags any mismatch as detector drift — the same "detector can be wrong" guard BOLA has. No real LLM, zero run-to-run drift.

Design decisions spec ambiguous

  • Controls are a harness default; the card can only DEGRADE or make them EFFICIENT. Technique cards don't carry oracle-control language, so keying discipline literally off card text would make a normal incumbent look undisciplined. Instead the researcher runs controls by default; an overclaim card turns them off (→ FDR), an efficiency card drops a redundant probe (→ measured cost cut).
